# Радар будущего

> [!summary] Суть
> Домашний бот для Telegram-канала **«Радар будущего»**: RSS → OpenAI → канал. Умная очередь, равномерная публикация, еженедельные рубрики (гаджеты, RSS-тренды, GitHub-сигналы). Управление: веб-панель + команды в личке бота.

**Не новостной агрегатор.** Приоритет: первоисточники, внедрения, слабые сигналы.

## Основной пайплайн и равномерная публикация

> [!important] Два cron
> При `publishEvenSpread: true` (по умолчанию):
> - **`postIntervalCron`** — только RSS + очередь (collect-only)
> - **`publishIntervalCron`** — публикация из очереди по графику

| Параметр | Типичное значение |
| --- | --- |
| `maxPostsPerDay` | 18 |
| `maxPostsPerRun` | 3 |
| `postIntervalCron` | `15 */6 * * *` |
| `publishIntervalCron` | `5 */2 * * *` |

Логика: `postsDueByNow()` — сколько постов «должно» быть к текущему моменту; за тик публикуется `min(отставание, maxPostsPerRun)`.

Файлы: `runPipeline.ts`, `runPublishTick.ts`, `publishFromQueue.ts`, `utils/evenPublish.ts`.

```mermaid
flowchart LR
    RSS[Cron RSS 6h] --> Queue[news.json]
    Pub[Cron publish 2h] --> Tick[runPublishTick]
    Tick --> Queue
    Tick --> TG[Telegram канал]
    Manual["/run"] --> Pipeline[runPipeline + publish]
```

- TTL по уровню, `finalScore`, min score, max 50 в очереди
- `/queue`, `/queue-prune`, `/source-stats`
- Архив: `data/queue-archive.json`

## Еженедельные рубрики

| Локальное время ПК | Рубрика | Команда | Cron `.env` |
| --- | --- | --- | --- |
| **Ср, Сб** 10:25 | 📦 Будущее в коробке | `/box` | `25 10 * * 3,6` |
| **Вс** 10:40 | 🔮 GitHub-сигналы | `/github` | `40 10 * * 0` |
| **Вс** 11:20 | 🧭 Направление недели | `/trends` | `20 11 * * 0` |
| **Вс** 19:00 | 🧩 Странный GitHub недели | `/weird` | `0 19 * * 0` |

> [!warning] Часовой пояс и минуты
> `node-cron` — **локальное время Windows**. Publish `:05`, RSS `:15`, рубрики на других минутах — без гонок `activeTask`.

### GitHub-сигналы (GitTrend)

| Кто | Когда | Что |
| --- | --- | --- |
| **GitTrend** (GitHub Actions) | **Сб 21:00 МСК** (18:00 UTC) | Пушит `reports/weekly-radar.json` |
| **Radar** | **Вс 10:40** локально | Скачивает JSON → посты в канал |

```text
Сб 21:00 МСК  →  GitTrend push JSON (авто, GitHub Actions)
Вс 10:40 МСК  →  Radar: trends[] → enrich → Telegram
Вс 11:20 МСК  →  Направление недели (RSS)
Вс 19:00 МСК  →  Radar: weirdFindOfTheWeek → Telegram
```

> [!note] Ручной push JSON
> Досрочная выкладка `weekly-radar.json` в GitTrend **не меняет cron** (сб GitTrend, вс Radar). Меняется только файл на GitHub; автопубликация — по воскресному расписанию или `/github` / `/weird`.

Цепочка трендов: `fetchWeeklyRadar` → `validateReport` → `selectTrendsForPublish` → `enrichGitTrend` → `buildGitTrendPost` → `sendPost`.

Цепочка странного: тот же JSON → `weirdFindOfTheWeek` → `buildWeirdGitHubPost` → `sendPost` (текст из GitTrend + `#GitHub`).

- 0–3 тренда/неделю, не дневной лимит
- Пустой `trends: []` — **не ошибка**, пост не делаем
- Фильтры: `GITTREND_MIN_SIGNAL_STRENGTH`, кулдаун категории 14 дн., неделя не повторяется (`gittrend.json`)
- **Анонс рубрики** (`gitTrendIntro.ts`) — только если `gittrend.json` → `published` **пустой** (первый пост серии); посты 2–3 всегда без анонса
- Расписание в анонсе: GitHub **10:40**, «Направление недели» **11:20**
- Текстовые посты до **4096** символов (`sendPost.ts`); enrich — развёрнутый `futureWhy`
- `/github force` — повтор недели
- **Странный GitHub:** поле `weirdFindOfTheWeek` в JSON (`telegramPost`, `weirdScore`); cron **вс 19:00**, `/weird`, state `gittrend-weird.json`; `/weird force`
- **Превью без канала:** `npx tsx scripts/preview-gittrend-admin.ts` (не трогает state)
- Подробности cron: `RADAR-SCHEDULE-UPDATE.md`

### Направление недели

3 направления из RSS-сигналов за 7 дней. Отдельно от GitHub.

- **Формат:** HTML (`parseMode: HTML`), как основные посты канала
- **Структура:** `🧭 НАПРАВЛЕНИЕ НЕДЕЛИ` → заголовок недели → summary → 3 блока (номер + название жирным, текст, **Горизонт:**) с разделителем `━━━━━━━━━━━━━━━━`
- **Длина:** цель **1500–2000** символов (summary и тренды по 2–3 предложения); макс. **4096**
- **Хэштег:** `#Science` (`hashtagForWeeklyTrends`)
- Нормализация горизонта: GPT может вернуть `1-3 года` → `1-3 years` (`generateWeeklyTrends.ts`)
- При переборе — мягкая подгонка описаний, не обрезка summary до 200 символов

### Лимиты Telegram по рубрикам

| Рубрика | Доставка | Генерация | API-лимит |
| --- | --- | --- | --- |
| Направление недели | текст HTML | 1500–2000 → fit 4096 | 4096 |
| GitHub-сигналы | текст HTML | до 4096 | 4096 |
| Странный GitHub | текст из GitTrend | `telegramPost` | 4096 |
| Будущее в коробке | фото + текст | до 2500 | текст 4096; caption 1024 → **split** |

> [!tip] Почему split в коробке
> Telegram не даёт подпись к фото длиннее **1024** символов. Рубрика шлёт фото без подписи и сразу полный текст вторым сообщением.

### Будущее в коробке

> [!note] Визуальная рубрика
> **Физическое устройство + фото** → `sendPhoto`. Приоритет реальным гаджетам, а не формальным полям AI.

**Источники** (`inTheBoxSources.ts`, приоритет `boxPriority`):

1. Engadget (категории Smartphones, Computing, …)
2. Tom's Guide (phones, laptops, smart-home, …)
3. T3 (tech, smart-home, wearables, …)
4. 3DNews, Mobile-Review (`/reviews/`, `/obzor-`)
5. Apple, Samsung, Meta, Nothing, Framework, …

**Формула допуска:**

```text
boxCandidate === true  +  image (RSS или og:image)  +  vision OK  →  sendPhoto + sendMessage
```

`whyThisIsADevice` — только для текста поста, **не** gate публикации.

**Этапы:**

1. Pre-filter — расширенный `DEVICE_HINT`, URL с `review`/`обзор` → сразу в AI
2. `articleImage.ts` — og:image / twitter:image / JSON-LD, если в RSS нет фото
3. AI batch (`analyzeGadget.ts`) — `boxCandidate`, score ≥ 6
4. Vision (`verifyDeviceImage.ts`)
5. Пост (`generateInTheBoxPost.ts`) — до **2500** символов; перебор кандидатов при сбое
6. Публикация: `splitPhotoAndText` — фото, затем полный текст (`sendPost.ts`)

**Стратегический запас** (`in-the-box-reserve.json`):

| | |
| --- | --- |
| Размер | **3** готовых поста (HTML + фото URL) |
| TTL | **7 дней** |
| Пополняет | **только cron** (среда/суббота) — лишние после публикации live |
| Берёт | **cron ср/сб** и **ручной `/box`**, если live RSS не вышел |

```mermaid
flowchart TD
    Run["/box или cron"] --> RSS[Свежий RSS]
    RSS --> Pub{Опубликован live?}
    Pub -->|да| ReserveAdd[Cron: лишние → запас]
    Pub -->|нет| Stock{Запас не пуст?}
    Stock -->|да| PubStock[Публикация из запаса]
    Stock -->|нет| Fail[Отчёт /boxstats]
```

**Расписание слотов:**

- Cron **среда** и **суббота** 10:25 — отдельные слоты
- Ручной `/box` **не блокирует** cron-слоты (учитывается только `trigger: cron` + `scheduledSlot`)

**Команды:** `/box`, `/boxstats`, `/boxreserve`

**Данные:** `in-the-box-rejections.json`, `in-the-box-stats.json`, `fromReserve: true` в `in-the-box.json`

**Маршрутизация:** не-устройства с `interestingForRadar` → основной Radar

**Формат поста:**

```text
📦 БУДУЩЕЕ В КОРОБКЕ
[фото устройства]

Заголовок
Что это: …
Что внутри: …
Почему это интересно: …
📡 Наблюдение: …
Источник / Ссылка
```

## Запуск (launcher)

Как у Gitrend / Jarvis:

```bash
npm run launcher:shortcut
```

| Файл | Назначение |
| --- | --- |
| `Radar Future.lnk` | Ярлык в корне — двойной клик |
| `launcher/Radar.ico` | Иконка радарного дисплея (кольца, луч, blips) |
| `launcher/Radar.vbs` | Скрытый старт без CMD |
| `launch-radar.ps1` | Build при необходимости, скрытый `npm start` |
| `data/launch.log` | Лог launcher |
| `data/server.log` | stdout/stderr бота |

Поведение: без залпа в канал (`RADAR_SKIP_INITIAL_PIPELINE`), cron и Telegram сразу после `/api/health` на `:3847`. Остановка: **`/stop`** в личке бота.

Опционально: `npm run launcher:build` → `launcher/Radar.exe` (pkg).

Пересоздать ярлык после обновления иконки: снова `npm run launcher:shortcut`.

## Запуск без залпа в канал

`npm start` при `paused: false` сразу гоняет `runPipeline` (до 3 постов в канал). Без залпа: **ярлык launcher** или `"paused": true` → `npm start` → `/resume`.
