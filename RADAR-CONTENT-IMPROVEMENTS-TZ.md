# ТЗ: улучшение качества контента канала «Радар будущего»

Цель — поднять редакторское качество постов основного потока, не ломая существующий пайплайн, скоринг, рубрики и расписание. Все правки точечные, обратносовместимые, конфигурируемые через `.env`.

Работаем только с основным потоком новостей (analyze → generate post). Рубрики (in-the-box, gittrend, weird, weekly trends) **не трогаем**.

---

## Правка 1. Вынести модели OpenAI в конфиг и поднять модель для текста поста

**Проблема.** И анализ, и генерация текста поста жёстко прибиты к `gpt-4o-mini`. Текст поста — это то, что видит читатель; на mini он выходит плоским.

**Что сделать.**

1. В `src/config.ts` добавить в `envSchema` две переменные:
   - `OPENAI_ANALYSIS_MODEL` — `z.string().default("gpt-4o-mini")`
   - `OPENAI_POST_MODEL` — `z.string().default("gpt-4o")`
2. В `src/ai/analyzeNews.ts` заменить хардкод `model: "gpt-4o-mini"` (строка ~255) на `config.OPENAI_ANALYSIS_MODEL`.
3. В `src/ai/generateTelegramPost.ts` заменить хардкод `model: "gpt-4o-mini"` (строка ~165, в `generatePostParts`) на `config.OPENAI_POST_MODEL`.
4. В `.env.example` и `README.md` (раздел `.env`) добавить обе переменные с дефолтами и одной строкой описания.

**Не делать.** Не менять модель у рубрик и vision-проверки. Не трогать `generateObserverComment` (там своя модель — оставить как есть).

**Критерий приёмки.** Без env-переменных поведение анализа не меняется; текст постов генерируется на `gpt-4o`. Обе модели переопределяются через `.env`.

## Правка 2. Сделать шаблон поста менее «бланочным»

**Проблема.** Каждый пост — одинаковая жёсткая форма (`Что произошло / Почему это важно / Горизонт / Сферы / Источник`). В ленте из ~18 постов в день это читается как отчётность.

**Что сделать.** Минимально инвазивный вариант (без переписывания `buildPost`):

1. В `src/ai/generateTelegramPost.ts`, блок «Что произошло» сделать опциональным в выводе: если `whatHappened` по смыслу дублирует заголовок, модель может вернуть его пустым, и тогда блок `<b>Что произошло:</b>` в `buildPost` **не выводится** (сейчас выводится всегда).
2. Тон по уровням должен различаться. В `SYSTEM_PROMPT` (англоязычный и RU-вариант) добавить инструкцию: для `breakthrough` — тон сдержанно-весомый, для `signal` — спокойно-наблюдательный, для `failure` — фактологичный без драматизации. Уровень уже передаётся в user-message (`LEVEL_DESCRIPTIONS`), нужно лишь явно попросить подстраивать тон.

**Не делать.** Не вводить несколько разных HTML-шаблонов — только условный вывод блока «Что произошло» и тональные нюансы. Структура поста и порядок блоков сохраняются.

**Критерий приёмки.** Если `whatHappened` пустой — в посте нет пустого заголовка-блока и лишних переносов строк. HTML остаётся валидным для Telegram (`parseMode: HTML`).

## Правка 3. «Почему это важно» должно смотреть вперёд (суть радара)

**Проблема.** Концепт канала — раннее обнаружение и слабые сигналы. Сейчас `whyImportant` пересказывает значимость в настоящем, а не отвечает на радарный вопрос «куда это ведёт / что предвещает».

**Что сделать.** В `src/ai/generateTelegramPost.ts`, в обоих системных промптах (`SYSTEM_PROMPT` и `SYSTEM_PROMPT_RU_SOURCE`) переформулировать требование к полю `whyImportant`:

- Должно содержать **взгляд вперёд**: к чему это ведёт, какой это сигнал о траектории технологии, что станет возможным/изменится, если тренд продолжится.
- Запрет на пустой пересказ настоящего и общие слова («это важный шаг», «это меняет индустрию» без конкретики).
- Привязка к переданному `impactHorizon`: формулировка должна быть согласована с горизонтом (now / 1–3 / 3–7 / 10+ лет).
- Длина — 1–2 предложения, без хайпа (тон уже задан).

**Не делать.** Не менять JSON-схему (`postResponseSchema`), названия полей и `buildPost`. Меняется только текст промпта.

**Критерий приёмки.** На тестовой выборке `whyImportant` формулирует следствие/траекторию, а не повтор `whatHappened`. Схема ответа не изменилась.

## Правка 4. Поднять ранний порог отбора

**Проблема.** `MIN_TRACK_SCORE = 5` в `src/ai/analyzeNews.ts` (строка ~166) пропускает в работу «середнячок» (5 из 10).

**Контекст (важно, не сломать).** Пороги **публикации** уже заданы отдельно в `config.ts` (`MIN_SCORE_SIGNAL/IMPACT/BREAKTHROUGH = 7`, `FAILURE = 8`). `MIN_TRACK_SCORE` — это более ранний гейт: что вообще попадает в `observations` и в очередь. То есть эта правка влияет на шум в наблюдениях и очереди, а не на финальную планку публикации.

**Что сделать.**

1. Вынести `MIN_TRACK_SCORE` в конфиг: `MIN_TRACK_SCORE` — `z.coerce.number().int().min(1).max(10).default(6)` (поднять дефолт 5 → 6).
2. В `analyzeNews.ts` использовать `config.MIN_TRACK_SCORE` вместо локальной константы.
3. Добавить в `.env.example` с комментарием.

**Критерий приёмки.** По умолчанию в работу не проходят items со score < 6. Значение переопределяется через `.env`. Логика per-level порогов публикации не затронута.

## Правка 5 (опционально, обсудить отдельно). Сшивка постов по теме

**Проблема.** Единственная нить между постами — блок «Сигнал подтвердился» (impact/breakthrough ↔ прошлое наблюдение). Радар по смыслу должен накапливать траекторию темы.

**Статус.** Это более крупная фича (нужна модель связей между постами и хранилище тем). **В рамках этого ТЗ не реализовывать** — вынесено как направление на будущее. Реализуется отдельной задачей после правок 1–4.

## Общие требования

- Сохранить обратную совместимость: без новых env-переменных проект собирается и работает как раньше (кроме сознательного апгрейда модели текста и порога — это и есть цель).
- `npm run build` проходит без ошибок TypeScript.
- Не трогать: рубрики, расписание/cron, скоринг очереди (`queueScore`), `sendPost.ts`, dashboard, хранилища.
- Промпты редактируются по смыслу описанного, формулировки — на усмотрение исполнителя, но JSON-схемы ответов остаются неизменными.

## Порядок работ

1. Правка 1 (модели в конфиг) — фундамент, делать первой.
2. Правки 3 и 2 (промпты: взгляд вперёд + тон/опциональный блок).
3. Правка 4 (порог).
4. Сборка `npm run build`, проверка `npm run dry` (генерация без публикации в канал).
