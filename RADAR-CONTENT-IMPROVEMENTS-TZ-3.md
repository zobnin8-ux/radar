# ТЗ-3: убрать унылость текста постов «Радар будущего»

Продолжение `RADAR-CONTENT-IMPROVEMENTS-TZ.md` и `-TZ-2.md`. ТЗ-1/2 поправили модели, форму, цвет уровней и наблюдателя — но **унылость осталась**, потому что не тронут главный её источник: **регистр прозы**. Посты пишутся абстрактно и в сослагательном наклонении, конкретные факты из источника теряются.

Это ТЗ — про текст тела поста. Всё точечно, обратносовместимо, без слома пайплайна, рубрик, расписания, скоринга. Работаем только с основным потоком (`generateTelegramPost` + фильтр наблюдателя). Рубрики, `sendPost.ts`, dashboard, хранилища — **не трогаем**.

Контекст: очередь `data/news.json` уже очищена вручную, миграция данных не нужна. Новые посты генерируются свежим кодом при публикации.

---

## Корневая проблема (зафиксировать перед правками)

Пример с реальной публикации:

> Источник (DeepMind): «cut application decision times by **50%**», правительство Великобритании.
> Пост: «это **может значительно сократить** время на разработку проектов».

Конкретика — **50%**, гос-внедрение — то, что цепляет читателя, — превращается в вату. И так почти в каждом посте: «может значительно…», «в долгосрочной перспективе…», «открывает дорогу…». Причина: системные промпты `generateTelegramPost` просят «1–2 предложения взглядом вперёд» и не требуют **ни одного конкретного факта из источника**. Модель по умолчанию выдаёт обобщённые гипотезы в сослагательном наклонении.

Цель ТЗ — сделать прозу **конкретной и фактологичной**: число / имя / масштаб / сравнение «было → стало», минимум сослагательного наклонения.

---

## Правка 1. Конкретика вместо абстракции (главное)

**Проблема.** `whatHappened` и `whyImportant` пишутся общими словами; факты (цифры, имена, проценты, сравнения) из заголовка/описания не попадают в пост.

**Что сделать.** В `src/ai/generateTelegramPost.ts`, в **обоих** системных промптах (`SYSTEM_PROMPT` и `SYSTEM_PROMPT_RU_SOURCE`):

1. Добавить жёсткое требование к конкретике:
   - Если в заголовке или описании источника есть **число, процент, срок, имя компании/продукта, масштаб или сравнение** — оно **обязано** появиться в посте (в `whatHappened` или `whyImportant`).
   - `whatHappened` должен **вести с самым конкретным/неожиданным фактом**, а не с общим пересказом. Пример: не «DeepMind применяет ИИ для планирования», а «ИИ DeepMind сократил сроки согласования строек в Британии вдвое (с гос-данными)».
2. Сменить регистр `whyImportant` с гипотетического на следственный:
   - Формулировать **следствие**, а не возможность: «к чему это ведёт», привязанное к конкретике, а не «это может в перспективе…».
   - Запретить как **основной** регистр сослагательное наклонение и слова-заполнители: «может значительно», «возможно», «потенциально», «в (долгосрочной) перспективе», «способно изменить», «открывает дорогу/возможности». Сослагательное допустимо изредка для честной неопределённости, но не как каркас фразы.
3. В `buildPostUserMessage` усилить подачу источника: явно пометить, что `Описание` — главный источник фактов, и просить выбрать из него **один ключевой факт** для поста. (Поле `Описание`/`Description` уже передаётся — менять состав не нужно, только акцент в инструкции промпта.)

**Не делать.** Не менять JSON-схему `postResponseSchema`, названия полей, `buildPost`, `selectLayout`. Не трогать правило дословного заголовка для RU-источников (`parts.headline = news.title`). Не вводить хайп/кликбейт — тон сдержанный (правила тона из ТЗ-1/2 сохраняются).

**Критерий приёмки.** На `npm run dry`: если в источнике есть число/имя/масштаб — оно есть в посте. `whyImportant` в большинстве постов сформулирован как следствие, а не «может…». Доля постов, начинающихся «Почему важно» со слов «может/возможно/в перспективе», близка к нулю.

## Правка 2. Достроить фильтр зачинов наблюдателя (закрыть протечку «Представьте»)

**Проблема.** ТЗ-2 добавил запрет зачинов «Представьте/Кажется…» **только в промпт** наблюдателя. В фильтре `CLICHE_PATTERNS` этих паттернов нет, поэтому, когда модель нарушает запрет, штампованный зачин всё равно проходит в канал.

**Что сделать.** В `src/utils/observerComment.ts`, в массив `CLICHE_PATTERNS` добавить паттерны на **начало строки**:

```js
/^\s*представьте/i,
/^\s*кажется[,\s]/i,
/^\s*похоже[,\s]/i,
/^\s*интересно[,\s]/i,
```

При срабатывании `shouldIncludeObserver` вернёт `false` → блок «Наблюдение» просто не выводится (публикация не падает).

**Не делать.** Не менять пороги `OBSERVER_MIN_WORDS`/`OBSERVER_MAX_WORDS` и логику overlap. Не трогать `OBSERVER_SYSTEM_PROMPT` (запрет там уже есть).

**Критерий приёмки.** Комментарий, начинающийся с «Представьте/Кажется/Похоже/Интересно», не публикуется. На `npm run dry` в блоках «Наблюдение» таких зачинов нет.

## Правка 3 (опционально). Поле keyFact в схеме — гарантия факта в посте

**Проблема.** Даже с усиленным промптом (правка 1) нет машинной гарантии, что факт реально попал в текст.

**Что сделать.** Если правки 1 недостаточно по результатам `dry`:

1. В `src/ai/generateTelegramPost.ts` расширить `postResponseSchema` **необязательным** полем `keyFact: z.string().optional()` — «самый конкретный факт из источника (число/имя/масштаб) или пустая строка, если фактов нет».
2. В промптах попросить заполнять `keyFact` и **обязательно использовать его** в `whatHappened`/`whyImportant`.
3. В коде: если `keyFact` непустой, но не встречается в тексте поста — залогировать `warn` (без падения). Поле в `buildPost` **не выводится** — оно только дисциплинирует модель.

**Не делать.** Не делать поле обязательным (обратная совместимость: старые ответы без `keyFact` должны парситься). Не выводить `keyFact` в пост отдельным блоком.

**Критерий приёмки.** Ответы без `keyFact` по-прежнему валидны. При наличии факта в источнике он отражён в посте; рассинхрон логируется.

## Общие требования

- Обратная совместимость: без новых env-переменных проект собирается и работает.
- `npm run build` проходит без ошибок TypeScript.
- JSON-схема меняется только в правке 3 и только **аддитивно** (необязательное поле); правки 1–2 схему не трогают.
- Не трогать: рубрики, `sendPost.ts`, cron/расписание, `queueScore`/`sourceScore`, dashboard, хранилища, пороги публикации.
- Промпты редактируются по смыслу; формулировки — на усмотрение исполнителя.

## Порядок работ

1. Правка 2 (фильтр зачинов) — одна строка списка, минимальный риск.
2. Правка 1 (конкретика и регистр) — основной эффект; редактируем оба промпта + акцент в user-message.
3. `npm run build`, затем 2–3 прогона `npm run dry` — оценить глазами: есть ли в постах цифры/имена/сравнения, ушло ли сослагательное «может…».
4. Если факт всё ещё теряется — правка 3 (`keyFact`).

## Как проверять результат (важно)

Унылость — это про регистр, её видно только на готовых постах. После сборки гонять `npm run dry` (генерация без публикации в канал) и читать вывод как ленту: каждый ли пост содержит конкретику, разнятся ли формулировки, нет ли возврата к «может значительно / в перспективе / Представьте».
